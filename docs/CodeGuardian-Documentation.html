About this document

This is the complete, self-contained reference for CodeGuardian AI for Laravel. It covers installation, every configuration key, all 20 artisan commands with their options and examples, the detection catalogue, reporting formats, the web dashboard, the safe refactoring pipeline, CI/CD integration, the AI engine, internal architecture, and troubleshooting. It is intended both as an onboarding guide for new users and as a submission-ready specification.

Section 1

Introduction & Overview

CodeGuardian AI is a static code analysis, security auditing, automated refactoring, and reporting platform for Laravel (PHP) and Flutter (Dart) projects. It installs as a Composer package and runs entirely through artisan commands or a built-in web dashboard.

No API key needed. No internet required. No cost. 100% embedded. The default static engine performs all core analysis offline and free. AI mode is an optional upgrade for richer, intent-aware explanations.

What problem does it solve?

Most teams discover architectural debt, security holes, and performance traps too late — in production, in incidents, or in painful code reviews. CodeGuardian shifts that discovery left: it scans the codebase deterministically, scores its health across six dimensions, explains every finding with a concrete fix, and can safely apply many fixes automatically behind a test-verified safety net.

The core workflow

  1. Analyze — scan code and produce findings, scores, and a risk rating.
  2. Generate tests — write PHPUnit stubs before any code changes.
  3. Refactor — apply deterministic + AI-driven improvements, one file at a time.
  4. Validate — re-run tests; auto-rollback any change that introduces a regression.
  5. Report — emit enterprise-grade reports (console, HTML, JSON, Markdown, SARIF, JUnit).

Who is it for?

Engineering teams

Continuous code-health gating in CI, PR review automation, and onboarding-quality reports.

Tech leads & architects

Six-dimension scorecards, module dependency graphs, and trend tracking over time.

Security engineers

OWASP/CWE-mapped findings, dependency CVE audits, and SARIF for code-scanning platforms.

Individual developers

A fast inner-loop watcher, deterministic auto-fixes, and a friendly local dashboard.

Section 4

Installation

Option A — Install from GitHub (any machine)

Step 1. Add the repository to your project's composer.json:

{
    "repositories": [
        {
            "type": "vcs",
            "url": "https://github.com/bilalmahoon/codeguardian-laravel"
        }
    ]
}

Step 2. Require the package:

composer require codeguardian/laravel:dev-main

Step 3. (Optional) Publish the config:

php artisan vendor:publish --tag=codeguardian-config

This creates config/codeguardian.php where you can customise skip directories, output paths, rules, gates, and more.

Option B — Local development (path repository)

If you have the package source on disk:

{
    "repositories": [
        {
            "type": "path",
            "url": "/absolute/path/to/packages/codeguardian-laravel",
            "options": { "symlink": true }
        }
    ]
}
composer require codeguardian/laravel:@dev

One-command setup

php artisan codeguardian:init                 # publish config + scaffold CI + (optional) git hook
php artisan codeguardian:init --ci=gitlab --hook

codeguardian:init publishes the config, writes a ready-to-use CI workflow (GitHub Actions or GitLab CI, auto-detected), and can install a pre-commit hook that runs a fast incremental scan. Nothing is overwritten without --force.

Verify the setup

php artisan codeguardian:doctor          # health check with fix suggestions
Section 5

Quick Start

The only required .env setting is — nothing. By default the tool runs in static mode, fully offline, with no API key.

# 1. Analyze the entire project
php artisan codeguardian:analyze

# 2. Analyze a specific area
php artisan codeguardian:analyze --path=app/Http/Controllers
php artisan codeguardian:analyze --module=User
php artisan codeguardian:analyze --api=orders

# 3. Analyze, then automatically FIX issues (safe, test-verified, auto-rollback)
php artisan codeguardian:analyze --fix

# 4. Security or performance only
php artisan codeguardian:security --fail-on=high
php artisan codeguardian:performance

# 5. Full interactive refactoring workflow
php artisan codeguardian:refactor

# 6. Open the dashboard
# https://your-app.test/codeguardian
When analyze finds issues in an interactive terminal, it offers to fix them right there (skipped automatically in CI / --no-interaction).

Section 8

What It Detects

AnalyzerWhat it finds
ArchitectureFat controllers (>150 lines), missing service layer, direct DB in controllers, inline validation, facade overuse, business/integration logic in models, env() used outside config/ (null after config:cache).
SecuritySQL injection, XSS {!! !!}, hardcoded secrets & API keys, mass assignment ($request->all() and $guarded = []), missing authorization, debug code, insecure uploads, command injection, code injection (eval), insecure deserialization, weak crypto (md5/sha1 on secrets), predictable randomness for tokens, path traversal, SSRF, open redirect, blanket CSRF exemption, hard-enabled debug, dynamic file inclusion, disabled TLS verification.
PerformanceN+1 queries, query-in-loop (whereIn batching), over-fetching (::all()->filter()), nested loops (O(n²)), Model::all() without pagination, count() on loaded collections, missing eager loading, exports without chunking.
Tech DebtLarge/God classes, high cyclomatic complexity, duplicated code blocks, TODO/FIXME debt, dead code, missing return types, magic numbers, deep nesting, long parameter lists, boolean-flag parameters, empty/swallowed catch blocks.
DatabaseIrreversible migrations (no down()), unindexed foreign keys, money as float/double, hard-to-evolve enum columns, missing unique indexes on lookup columns, fully unguarded models.
Flutter / Dartprint() in production, setState() inside build(), oversized build() methods, BuildContext used across an async gap.
Test GeneratorPHPUnit stubs from method signatures (controller, service, model, generic).
RefactoringAuto-fixes ($request->all() → validated(), remove dd()/dump()); reports manual fixes for the rest.
Security taxonomy. Every security finding is mapped to its OWASP Top 10 (2021) category and a CWE id, with an indicative CVSS band, a confidence level, expected impact, estimated effort, and breaking-change risk — so findings are triage-ready.

Anatomy of a finding

Every finding carries the context an engineer needs to triage and fix it:

FieldMeaning
severitycritical / high / medium / low
confidencehigh / medium / low — how sure the rule is
categorymachine-readable type, e.g. sql_injection, n_plus_one, god_class

root_causethe underlying reason it happens
code_snippetthe evidence (the offending line)
recommendationthe suggested fix
code_before / code_aftera concrete better implementation
impactthe expected benefit of fixing it
efforttrivial / small / medium / large
breaking_risknone / low / medium / high
cwe / owaspsecurity taxonomy
principlethe engineering principle involved (SOLID:SRP, Clean Code, …)

Risk score (explainable)

Alongside the quality scores (how good the code is), analyze produces a risk score (0–100, how urgently it needs attention), weighted by severity × confidence, with reasoning:

  Risk Score: 62/100  (CRITICAL)
    • 2 critical finding(s) drive the risk to its ceiling — these should block release.
    • Concentrated in: sql injection (2), n plus one (3), missing types (4).
    • Weighted across 14 finding(s) by severity × confidence.

Section 12

Web Dashboard

Prefer a UI over the terminal? CodeGuardian ships a built-in dashboard. After installing the package, open https://your-app.test/codeguardian.

What you can do

  • Run everything from the browser — analyze, security audit, performance review, generate tests, and refactor. Targets are chosen from searchable dropdowns (modules, API routes, web routes, artisan commands).
  • Watch live progress — each run executes in the background and streams its console output to the page in real time.
  • Browse full history — every past run with its type, target, status, and saved HTML/JSON report; the history page shows a project-health snapshot (score, risk, open issues, trend).
  • Explore findings interactively — a completed analyze run gets a Findings Explorer: filter by severity, search by title/file/category, and read each issue with its location and recommendation.
  • Track trends on the Insights page (/codeguardian/insights) — quality-score and risk trend charts, severity and top-category breakdowns, quality dimensions, and hotspot files.
  • Fix from the UI — a completed analyze run shows a "Fix these issues" panel. Select specific files or fix the whole scope; either way it launches the same safe, test-verified refactor.

Security

The dashboard is local-only by default (reachable only when APP_ENV=local). To control access:

  • Define a Gate named viewCodeGuardian in your AuthServiceProvider (takes precedence; restrict by user/role), or
  • Set CODEGUARDIAN_DASHBOARD_LOCAL_ONLY=false to open it (combine with the gate or your own middleware for production).
'dashboard' => [
    'enabled'           => env('CODEGUARDIAN_DASHBOARD', true),
    'path'              => env('CODEGUARDIAN_DASHBOARD_PATH', 'codeguardian'),
    'middleware'        => ['web'],
    'restrict_to_local' => env('CODEGUARDIAN_DASHBOARD_LOCAL_ONLY', true),
];

Customise the UI by publishing the views: php artisan vendor:publish --tag=codeguardian-views.

Section 13

Foolproof Refactoring

Refactors run a test → refactor → verify loop so a change that breaks a test never ships:

  1. Generate test stubs for the files in scope.
  2. Establish a baseline (pre-existing failures are ignored — only NEW failures count).
  3. Refactor each file (static auto-fixes + AI deep refactor).
  4. Re-run tests; if a file's refactor introduces a new failure, it is automatically rolled back to the original.

Enabled automatically from the dashboard, with --safe, in --mode=auto, or via config (refactor.safe_mode, refactor.auto_rollback_on_fail).

The full interactive workflow

1. Analyze code     → find all issues
2. Show report      → ask confirmation to proceed
3. Generate tests   → write test stubs BEFORE changing any code
4. Run tests        → establish baseline (passes/fails)
5. Refactor         → fix one file at a time (ask per file in interactive mode)
6. Run tests again  → verify nothing broke (rollback / continue / stop on failure)
7. Final report     → before/after summary saved to storage/codeguardian/reports/

Targeting

php artisan codeguardian:refactor --module=Order
php artisan codeguardian:refactor --api=v1/auth/login --safe
php artisan codeguardian:refactor --file=app/Services/AuthService.php
php artisan codeguardian:refactor --files="app/Services/AuthService.php,app/Http/Controllers/Auth/LoginController.php" --safe

--files= / --file= trace each file's dependency chain (via Reflection) and refactor them together.

Deterministic fixes (no AI)

php artisan codeguardian:fix --dry-run    # preview the safe mechanical fixes
php artisan codeguardian:fix              # apply (keeps a .cgbak backup per file)

codeguardian:fix applies only deterministic, rule-based transforms; every change is syntax-validated before writing, so the engine never produces broken PHP.

Section 14

Filtering & Suppression

Filters (change what's shown)

Filters are AND-combined; each CSV value is OR-combined.

FlagExampleKeeps
--severity--severity=critical,highonly those severities
--min-severity--min-severity=highat or above that severity
--category--category=sql_injection,n_plus_onecategories matching any substring
--confidence--confidence=highfindings at that confidence
--owasp--owasp=A03,A01security findings in those OWASP categories
--cwe--cwe=CWE-89findings with that CWE id

Suppression (remove entirely — from scores, reports, CI exit codes)

// config/codeguardian.php
'ignore' => [
    'categories' => ['magic_numbers', 'commented_code'],
    'paths'      => ['database/migrations/', 'tests/*'],
],

Inline, right in the source:

$hash = md5($value);                 // codeguardian-ignore weak_cryptography
// codeguardian-ignore-file          # suppresses every finding in this file

A bare // codeguardian-ignore suppresses any finding on that line; adding category names limits it. Use --no-suppress to temporarily see everything.

Section 15

Rule Configuration

Take full control of the engine from config — no code changes. Rules are keyed by their finding category; list them all with php artisan codeguardian:rules.

// config/codeguardian.php
'rules' => [
    'magic_numbers' => false,        // turn a rule off entirely
    'missing_types' => 'low',        // downgrade severity
    'n_plus_one'    => 'critical',   // upgrade severity
    'todo_debt'     => ['enabled' => true, 'severity' => 'low'],
],
php artisan codeguardian:rules                       # full catalog + effective state
php artisan codeguardian:rules --group=security      # one analyzer group
php artisan codeguardian:rules sql_injection         # full docs for one rule

Severity overrides apply before scoring, filters, and CI gates.

Tuning presets

php artisan codeguardian:analyze --preset=strict    # maintainability debt first-class
php artisan codeguardian:analyze --preset=lenient   # mute the noisiest low-value rules

Custom rules (no PHP)

Define project-specific detection rules in config('codeguardian.custom_rules'). Each rule scans file contents with a regex and emits a finding (severity, message, fix, path filters). They flow through scoring, filters, suppression, and CI gates like built-in rules.

Per-rule documentation

Every rule ships with a short explainer — what it means, why it matters, how to fix it, and authoritative references (OWASP / CWE / Laravel docs). HTML reports link each finding to its docs via a "Learn more" link.

Section 16

Continuous Integration

SARIF — GitHub / GitLab / Azure

--format=sarif emits schema-valid SARIF 2.1.0 with security-severity, stable partialFingerprints, CWE/OWASP helpUris, and one-click fixes[] where a concrete replacement exists.

# GitHub Actions
- run: php artisan codeguardian:analyze --format=sarif --plain
- uses: github/codeql-action/upload-sarif@v3
  with:
    sarif_file: storage/codeguardian/reports

Baseline / diff — fail only on new findings

# once, on a clean main branch — commit the file
php artisan codeguardian:analyze --write-baseline

# in CI on every PR — non-zero exit only if NEW findings appear
php artisan codeguardian:analyze --against-baseline --new-only --plain

Fingerprints are line-number-independent (category + file + title + normalised snippet). The summary reports new / existing / fixed counts.

Quality gates / budgets

'gates' => [
    'max_critical' => 0,
    'max_high'     => 5,
    'max_total'    => 200,
    'max_risk'     => 60,   // 0–100
    'min_score'    => 70,   // 0–100
    'min_quality'  => 70,   // composite 0–100
],

Any breach prints the violated budgets and exits non-zero.

Other CI integrations

  • Inline PR annotations — --annotate (auto in GitHub Actions) renders findings on the "Files changed" view, no upload.
  • PR / MR comment — codeguardian:comment posts a Markdown summary (auto-detects GitHub/GitLab; updates its previous comment).
  • JUnit panels — --format=junit lights up native test-report UIs.
  • Dependency audit — codeguardian:audit --fail-on=high scans composer.lock for CVEs and abandoned packages.
  • Doctor preflight — codeguardian:doctor --json returns non-zero on hard failures.
  • Test impact — codeguardian:test-impact --filter emits a PHPUnit filter to run only affected tests.

Exit codes: 0 = success (no issues at/above --fail-on); 1 = issues found at the specified level / gate breached / hard error.

Gemini (Google)gemini-1.5-flashFree tier available; fast.

Cost controls

  • AI response cache — identical prompts are served from disk for $0 (model is part of the key).
  • Diff-only review — codeguardian:review sends only changed lines, far cheaper than re-reviewing whole files.
  • Usage meter — every AI run prints token usage and estimated cost (💰 AI usage: … ~$0.0123); tune the price table via config('codeguardian.pricing').
Static mode finds the same structural issues as AI mode for most common patterns. AI mode adds natural-language summaries and catches subtler logic issues that require understanding program intent. If a key is missing or a model 404s, AI commands fall back to static.

Section 19

Extending the Engine

Adding a rule is a localized change to one analyzer:

  1. Open the relevant analyzer in src/Analyzers/.
  2. Add a private function checkYourRule(string $filePath, string $content): void and call it from analyze().
  3. Emit findings via AnalysisResult::make() with full metadata.
  4. Add a unit test in tests/Unit/Analyzers/ — positive fixture finds it, clean fixture does not.
$this->addResult(AnalysisResult::make(
    category:       'my_rule',
    severity:       'high',
    title:          'Short, specific title',
    description:    'Why it matters, with the line number.',
    file:           $filePath,
    lineStart:      $lineNum + 1,
    recommendation: 'The concrete fix.',
    confidence:     'high',
    impact:         'What fixing it buys you.',
    effort:         'small',
    breakingRisk:   'low',
    cwe:            'CWE-123',        // security rules
    owasp:          'A01:2021-...',   // security rules
    principle:      'SOLID:SRP',      // design rules
));
Design principle for new rules: high confidence, low false positives. When a pattern is ambiguous, gate it on a taint/context signal and set confidence accordingly rather than emitting a noisy finding.

For most teams, custom regex rules in config (Section 15) are enough — no PHP required.

Section 21

Frequently Asked Questions

Does it need an API key?

No. The default static engine runs fully offline with zero cost. An API key only unlocks the optional hybrid/ai modes.

Will it modify my code during analysis?

No. analyze, security, and performance are strictly read-only. Only refactor and fix write — and refactor runs a test-first safety net with automatic rollback.

How do I reduce noise?

Use filters (--min-severity=high, --confidence=high, --category=) or suppression (config + inline comments).

How do I run only part of the analysis?

--agents=security,performance, or the dedicated codeguardian:security / codeguardian:performance commands.

Can I add my own rules?

Yes — config-based custom regex rules (no PHP), or a small method on an analyzer plus a unit test.

Which Laravel / PHP versions are supported?

PHP ^8.1 and Laravel 9 / 10 / 11.

How do I make CI fail on regressions only?

Record a baseline once, then run --against-baseline --new-only --fail-on=high on PRs.
